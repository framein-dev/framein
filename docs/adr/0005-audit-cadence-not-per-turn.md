# ADR-0005 — 감사(리뷰)는 매 턴이 아니라 게이트·이상징후 트리거

- **Status:** Accepted (2026-06-20)
- **관련:** PRD v0.6 §5.8, §5.2(F-ROLE-3 쿼터), §5.5(F-ROUTE-3 "그만 제안") · [ADR-0001](./0001-zero-friction-first.md)

## Context

교차 에이전트 **감사**(reviewer 역할 / `/codex`형 호출)를 기본 기능으로 둘지 검토했다. 가치는
분명하다 — 한 LLM이 **잘못된 루프에 갇히거나 잘못된 가정으로 치닫는 것**을 잡고, 소스 품질을
올린다. 그래서 "항상 의견을 받을 수 있으면" 좋다.

하지만 **매 턴 감사를 디폴트로 켜면** 세 가지 문제가 생긴다.

1. **시어머니 효과** — 모든 작업에 시시콜콜 잔소리가 붙어 흐름이 끊긴다.
2. **속도** — 작업이 (2배까지는 아니어도) 훨씬 느려진다.
3. **비용/쿼터** — 토큰·쿼터가 사실상 2배로 탄다. 이는 framein이 내세우는 **쿼터 인식 전환**
   철학(F-ROLE-3)과 정면충돌한다.

핵심 통찰: 감사 가치는 **턴마다 균일하지 않다.** 대부분의 턴(파일 읽기, 변수 rename, 소소한
편집)은 둘째 의견이 필요 없고, 소수의 고레버리지 순간(랜드 직전, 설계 결정, 파괴적 작업,
**에이전트가 헛도는 순간**)만 필요하다. 따라서 결정은 on/off가 아니라 **cadence(언제) + signal
quality(무엇만)** 의 문제다.

## Decision

**감사는 매 턴이 아니다.** 다음으로 설계한다.

- **게이트 트리거(F-AUDIT-2):** 커밋/랜드 직전, ADR(설계 결정) 생성, 파괴적·되돌리기 어려운
  작업 직전에만 reviewer 호출. (사람 팀이 PR에서 리뷰하지, 키 칠 때마다 안 하는 것과 동일.)
- **이상징후 트리거(F-AUDIT-3):** 스토어/task-ledger의 **thrash 신호**(같은 파일 N회 재수정,
  동일 테스트 반복 실패, K턴 무진전, 순환 추론)를 감지하면 그때만 호출. 이는 "잔소리"의
  **정반대** — 평소 침묵, 삼천포로 빠질 때만 개입. *사용자가 원한 핵심 가치를 가장 잘 만족.*
- **블로커만(F-AUDIT-4):** 고신뢰 **P1만** 표면화, 나머지는 침묵. 신뢰는 침묵에서 나온다.
  반복 거부한 종류는 "그만 제안" 기억(F-ROUTE-3)으로 억제.
- **비동기·비차단(F-AUDIT-5):** diff/결정에 대해 백그라운드로 돌리고 결과를 주석으로 부착
  (수용/무시). 랜드 게이트만 블로킹 → 작업 시간 2배 방지.
- **다이얼(F-AUDIT-6):** `off → gates → gates+anomaly(기본) → paranoid(매 턴)`. 기본값은
  **gates+anomaly · blocker-only · async**.

## Consequences

- reviewer 역할은 **조용함으로 신뢰를 번다.** 감사 예산(토큰·쿼터)은 값어치 있는 순간에 집중.
- 구현 필요: 스토어/ledger 위의 **이상징후 감지기**, **비동기 리뷰 배관**, **P1/P2 신뢰도
  게이팅**. 임계값(재수정 횟수·무진전 턴 수·블로커 신뢰도)은 실측 튜닝(PRD §11.8).
- 마찰 제로([ADR-0001])·쿼터 인식(F-ROLE-3)과 정합.

## Alternatives considered

- **매 턴 항상 켜기** — 기각. 잔소리·속도·쿼터 2배. 마찰 제로 위반.
- **수동 호출만(기본 off)** — 기각. 사용자가 원한 "헛도는 루프 차단"을 놓침. 이상징후 트리거가
  그 가치를 자동으로 주되 잔소리는 피한다.
