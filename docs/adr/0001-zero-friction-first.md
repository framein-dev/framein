# ADR-0001 — 마찰 제로(Zero-Friction)를 제1 설계 제약으로

- **Status:** Accepted (2026-06-20)
- **관련:** PRD v0.4 §2.4, §5.7 · [ADR-0002](./0002-reuse-existing-mcp-and-skills.md)

## Context

Framein이 경쟁하는 진짜 기준선은 다른 멀티 에이전트 도구가 아니라 **"그냥 각 CLI(Claude/
Codex/Gemini)에 직접 타이핑하는 것"** 이다. 우리가 셋업 단계, 새 문법, 프롬프트 이스케이프,
파일 덮어쓰기 같은 **새 마찰을 더하는 순간** 사용자는 도구를 버리고 원래 방식으로 돌아간다.

내부 DX 리뷰에서 다음 마찰원이 드러났다.

- 작업 지시의 핵심 동사(`frame ask "..."`)가 요구사항에 빠져 있고, 프롬프트를 셸 인자로
  넘기면 따옴표·백틱·`$`·줄바꿈 이스케이프가 지옥이다.
- `init`/`sync`가 기존에 손으로 쓴 `CLAUDE.md` 등을 통째로 덮어쓴다.
- MCP 서버 기동·세 CLI 등록을 수동으로 해야 한다.

기준선은 분명하다: **"지금 Claude Code로 대화하며 개발하는 경험."** 그만큼 매끄러워야 한다.

## Decision

**새 사용자 마찰을 0에 가깝게 두는 것을 모든 기능 결정의 상위 제약으로 삼는다.** 기능의
넓이보다 우선한다. 구체적으로:

1. **셋업은 한 번, 명령 하나** — `npx framein init`이 스토어·세 CLI MCP 등록·컨텍스트 파일을
   자동 배선. 사용자가 서버를 직접 띄우거나 CLI별 설정을 손으로 쓰지 않는다.
2. **새 문법 학습 0** — 평소처럼 각 CLI를 그대로 쓰고 Framein은 밑에서 동기화만 한다(비가시성
   우선). 프롬프트 입력은 stdin/`$EDITOR`/REPL로 받고 셸 인자를 강제하지 않는다.
3. **기존 자산 보존** — managed-block만 삽입/갱신하여 사용자 작성 내용을 보존하고, 기존
   MCP·스킬은 상속·연합한다.
4. **명령은 소수·자명**, 파괴적 동작은 `--dry-run`/diff로 미리 보여준다.
5. **측정 가능한 게이트** — time-to-first-value, 셋업 단계 수를 릴리스 게이트 지표로 둔다.

## Consequences

- 셋업/프롬프트당 마찰을 더하는 기능은 **연기하거나 opt-in**으로 둔다(예: 스킬 자동주입
  라우터 → 후순위).
- 상호작용 모델(§5.7), managed-block 투영(§6.2), MCP 등록 자동화(§6.3)가 **필수 요구사항**이
  된다.
- 성공 지표에 마찰 측정치를 추가한다(PRD §10).
- 트레이드오프: 자동 배선·비가시성은 구현 복잡도를 높이지만, 채택률의 핵심 레버이므로 감수한다.

## Alternatives considered

- **기능 우선, 마찰은 나중에** — 기각. 마찰은 채택 자체를 막는 1차 리스크(PRD §9)라 "나중"이
  오지 않는다.
- **풍부한 CLI 플래그로 모든 걸 노출** — 기각. 외울 게 늘어 마찰 증가. 소수·자명 원칙과 충돌.
